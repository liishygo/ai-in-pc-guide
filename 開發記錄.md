## 一、初始需求

媒體工作者教學網頁，介紹 Google Antigravity，內容包含：
- 軟體安裝
- 操作介面介紹
- 安裝擴充模式
- 應用案例

風格參考：https://darkschen.github.io/n8n-learning-site/

## 二、目標對象重新定義（重要轉折）

**原始假設**：一般使用者 / 工程師取向

**修正後**：媒體內容生產者（非工程師）
- 媒體編輯、文字記者、影音企劃
- 核心需求：寫文稿、產圖提示詞、影片腳本、社群貼文

**影響**：整個教學架構需大翻新，語言要更貼近媒體工作者的日常，避免工程術語。

## 四、各區塊的內容決策

### Hero 區
| 決策 | 原因 |
|------|------|
| 刪除「這是一份專為媒體人設計的完整上手指南」| 太誇大，不實在 |
| 刪除「從問 AI 問題升級到讓 AI 在你電腦裡幹活」| 與標題概念重複 |
| 刪除三個概念卡片（從聊天到共事、不只用工具還能造工具、你指揮它執行）| 看不出與前面概念有何不同，內容冗餘 |

### 比較表（瀏覽器 AI vs 電腦端 AI）
| 項目 | 修改決策 | 原因 |
|------|---------|------|
| 「累積性」→「工作模式」| 標籤與內容不符，「工作模式」更準確 |
| 「AI 持續在你電腦上執行任務」→「AI 可以在你的電腦裡執行任務」| 語氣較精確，「持續」過度誇大 |
| 「每次對話都重來」改寫 | 瀏覽器 AI 其實同一對話可以連續聊，需要更強的論點 |
| 「規則每次要重講」改寫 | ChatGPT / Gemini 網頁版已有自訂指令功能，這個理由不夠強 |
| 刪除「敏感草稿不外流」| 不準確：改稿時草稿仍需傳至雲端 AI 處理 |
| 「不受檔案大小限制」→「檔案很多時，會很方便」| 前者有點誤導，後者較實際 |
| 刪除「瀏覽器 AI 像問顧問意見；電腦端 AI 像雇了一個人在你電腦裡幹活」| 刪除 |
| 「電腦端 AI（Antigravity）」→ 加入 Claude cowork、Claude code | 讓讀者知道不只一個工具 |

**工作模式核心論點（確認後的版本）**：
- 瀏覽器 AI：以對話為核心，每個專案只有一組預設指令，主要優化對話輸出品質
- 電腦端 AI：以工作流程為核心，制訂規則、固定流程、技能、工具等等，涵蓋對話、計畫、執行、初步成果、修改、最終結果等多個階段

### 操作介面區
- 原有「開啟文稿 → 對話下指令 → AI 寫入編輯區」箭頭流程圖：**刪除**（內容已在介面說明中涵蓋，重複）
- 強調三區塊：左側檔案總管 / 中間編輯區 / 右側 AI 對話區

### Markdown 區（新增）
內容定調：
- 用 AI 的人一定要認識 Markdown
- AI 最愛讀的格式：純文字、檔案小、能設定標題列點重點
- 餵 Markdown 給 AI 可大幅節省 token 消耗
- Word / PDF 要給 AI 分析，先轉成 Markdown

刪除：「Antigravity 原生支援，有即時預覽」← 無法查證，且 VS Code 的 Markdown 預覽需手動開啟，非真正即時

### 延伸模組區
**最終推薦清單**（從 6 個縮減為 4 個）：
| 延伸模組 | 用途 |
|---------|------|
| Chinese (Traditional) Language Pack | 介面中文化 |
| Mark Sharp | Markdown 增強 |
| Office Viewer | 直接在編輯器內預覽 Office 檔案 |
| Paste Image | 截圖直接貼進文稿，自動存圖 |

**特殊安裝說明（重要）**：
Chinese (Traditional) Language Pack 因版本關係無法直接從市集安裝，需：
1. 前往 open-vsx 下載套件包：https://open-vsx.org/extension/MS-CEINTL/vscode-language-pack-zh-hant/1.106.0
2. 三個點點 → Install from VSIX → 重新開啟 Antigravity

**安裝步驟新增第 2 步**：更改 Marketplace 網址
- Marketplace Item URL：`https://marketplace.visualstudio.com/items`
- Marketplace Gallery URL：`https://marketplace.visualstudio.com/_apis/public/gallery`

### 應用情境區
- 名稱：「應用案例」→「應用情境」
- 移除：入門難度標籤、複製按鈕
- 最終 3 個情境：

**情境 1：讓 AI 認識你**
建立 3 個 .md 檔放在專案資料夾：
- 關於我.md
- 我的寫作風格.md
- 參考資料.md（自己的稿件、常用資料來源等等）

**情境 2：Threads 貼文助手**
資料夾規劃：
- 參考資料：專家分享的貼文寫法、想模仿的貼文等等
- 寫稿：每篇貼文草稿、修改後的版本

工作流程：跟 AI 對話，建立寫稿流程、改稿流程 → 丟主題，AI 依照流程產出成品

**情境 3：根據書本與 AI 對話**
- AI 的依據：書與你的資料
- 對話時依照自己的狀況，從書中提取有用的行動建議

## 六、可複用的開發原則

1. **先定對象，再寫內容**：目標對象不同，架構完全不同。工程師 vs 媒體工作者需要的切入點截然不同。

2. **「能幫我做什麼」優先**：非工程師讀者需要先看到實際應用，再學操作設定。應用情境應盡早出現。

3. **說法要能查證**：功能說明若無法確認（如「原生支援即時預覽」），寧可刪除或改成保守說法。

4. **比較要公平**：若競品已有類似功能（如瀏覽器 AI 的自訂指令），不能用過時的弱點來比較，需找到真正的差異點。

5. **刪比加好**：每次修改大多是刪內容，而非加內容。冗餘的概念重複、誇大的宣告、無法查證的功能，一律刪除。

6. **特殊安裝步驟要特別標示**：像 Chinese Language Pack 無法直接安裝這類坑，必須在教學中明確說明，否則讀者第一步就卡住。
